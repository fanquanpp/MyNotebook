## 1. 为什么需要模块化 (Why Modules)

模块化的目标：

- 隔离作用域，避免全局变量污染
- 组织代码与依赖，提高可维护性
- 支持复用、测试、按需加载与打包优化

## 2. CommonJS（Node.js 传统模块）

### 2.1 核心语法

```js
// add.cjs
function add(a, b) { return a + b }
module.exports = { add }

// main.cjs
const { add } = require('./add.cjs')
console.log(add(1, 2))
```

特性要点：

- `require()` 是运行时加载（可以写在条件分支中）
- 导出对象是可变引用：`module.exports` / `exports`
- 循环依赖时拿到的可能是“未完成初始化”的导出对象（半成品）

### 2.2 CommonJS 循环依赖提示

避免在模块顶层立即执行依赖模块的逻辑；把依赖使用延迟到函数内部，降低循环依赖的初始化风险。

## 3. ES Modules（ESM，浏览器与现代 Node）

### 3.1 具名导出/导入 (Named)

```js
// math.js
export function add(a, b) { return a + b }
export const PI = 3.14159

// main.js
import { add, PI } from './math.js'
console.log(add(1, 2), PI)
```

### 3.2 默认导出/导入 (Default)

```js
// logger.js
export default function log(msg) { console.log(msg) }

// main.js
import log from './logger.js'
log('hello')
```

实践建议：

- 默认导出适合“一个模块一个核心能力”的场景
- 工程协作中更推荐具名导出，便于重构与自动补全

### 3.3 `import * as ns`

```js
import * as math from './math.js'
console.log(math.add(1, 2))
```

### 3.4 动态导入 (Dynamic import)

动态导入返回 Promise，适合按需加载与拆包：

```js
async function loadFeature() {
  const mod = await import('./feature.js')
  mod.run()
}
```

## 4. ESM vs CommonJS（关键差异）

| 维度 | CommonJS | ES Modules |
| :-- | :-- | :-- |
| 加载时机 | 运行时 | 静态分析 + 运行时 |
| 语法位置 | 可在任意位置 `require` | 顶层 `import/export`（动态导入除外） |
| 导出绑定 | 值拷贝/对象引用（可变） | 实时绑定（live binding） |
| 生态 | Node 传统 | 浏览器/现代 Node/打包器 |

## 5. Node.js 中的 ESM 实践

常见做法（择一）：

- `package.json` 设置 `"type": "module"`，`.js` 视为 ESM
- 使用 `.mjs` 后缀明确为 ESM
- 继续用 `.cjs` 保持 CommonJS

互操作注意：

- ESM 导入 CommonJS：`import pkg from 'cjs-pkg'` 常见会把 `module.exports` 映射到 default
- CommonJS 引入 ESM：需要动态 `import()`（或使用构建工具转换）

## 6. 工程化与最佳实践 (Tooling & Best Practices)

- 保持导出入口稳定：在模块根部集中导出（barrel export）需谨慎，避免循环依赖
- 避免“隐藏依赖”：不要在模块导入时产生副作用（例如顶层发请求/改全局状态）
- 对外 API 尽量使用具名导出，便于 Tree-shaking 与重构
