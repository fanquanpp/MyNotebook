## 1.1 GitHub

**名称**：GitHub（全球最大代码托管平台）

**首次出现位置**：C01_101-账户与安全.md 第1章

**定义**：
GitHub 是全球最大的代码托管平台和开源社区，由 Microsoft 收购后继续独立运营。它提供 Git 仓库托管、Issue 跟踪、Pull Request 协作、Actions 自动化等功能，是现代软件开发不可或缺的工具。

**详解**：
作为代码托管平台，GitHub 不仅存储代码，还管理着项目协作、Issue 跟踪、CI/CD 配置等关键信息。开发者可以在 GitHub 上创建公开或私有仓库，通过 Fork 和 Pull Request 机制参与开源项目，或进行团队协作开发。

**相关链接**：

- GitHub 官网：<https://github.com>

---

### 1.2 Repository（仓库）

**名称**：仓库（Repository）

**首次出现位置**：C01_103-仓库创建与管理.md 第1章

**定义**：
Repository（仓库）是 GitHub 上的项目容器，用于存储代码文件、提交历史、分支、Issues、Pull Requests、Actions 等多种协作功能和数据。

**详解**：
仓库分为三种类型：Public（公开仓库）对所有人可见；Private（私有仓库）仅对指定用户可见；Internal（内部仓库）仅对组织成员可见，需要 GitHub Enterprise。每个仓库都有唯一的 URL 地址，格式为 `https://github.com/用户名/仓库名`。

---

### 1.3 Commit（提交）

**名称**：提交（Commit）

**首次出现位置**：C01_103-仓库创建与管理.md 第5章

**定义**：
Commit 是 Git 中的基本操作单位，表示对工作区和暂存区变更的快照。每次提交都会记录修改的内容、提交者信息、时间以及父提交指针。

**详解**：
每个提交包含：提交信息（描述本次修改内容）、作者信息（提交者的姓名和邮箱）、日期时间、指向父提交的指针、以及包含文件快照的树对象。提交是 Git 版本控制的核心概念，通过提交可以追溯代码的历史变更。

---

### 1.4 Branch（分支）

**名称**：分支（Branch）

**首次出现位置**：C01_103-仓库创建与管理.md 第5章

**定义**：
Branch（分支）是指向特定提交的指针，允许开发者在独立的环境中开发新功能或修复 bug，而不影响主分支的稳定性。默认分支通常为 main 或 master。

**详解**：
分支操作包括：创建分支（git branch）、切换分支（git checkout）、合并分支（git merge）、删除分支（git branch -d）等。Git 的分支系统非常轻量，创建和切换分支几乎是即时完成的，这使得并行开发变得简单高效。

---

### 1.5 Fork（派生）

**名称**：派生（Fork）

**首次出现位置**：C01_103-仓库创建与管理.md 第7章

**定义**：
Fork 是在 GitHub 上创建他人仓库的副本的操作。派生后的仓库完全独立于原仓库，可以在不影响原项目的情况下进行修改和实验。

**详解**：
Fork 广泛应用于开源项目贡献。开发者首先 Fork 项目到自己的账户下，在副本中进行修改和测试，最后通过 Pull Request 将更改提交给原项目审核。Fork 功能使得任何人都可以参与开源项目，而不需要直接获取原项目的写权限。

---

### 1.6 Pull Request（拉取请求）

**名称**：拉取请求（Pull Request）

**缩写**：PR

**首次出现位置**：C01_102-SSH与HTTPS.md 第3章

**定义**：
Pull Request 是 GitHub 上用于代码审查和合并的机制。当开发者完成某个分支的修改后，可以发起 Pull Request 请求将代码合并到主分支。

**详解**：
Pull Request 提供了完整的代码审查流程：创建 PR 后，团队成员可以查看代码变更、进行评论、提出修改建议。PR 可以设置分支保护规则，要求通过 CI 检查和至少一个审查才能合并。这是一种安全可控的代码集成方式。

---

### 1.7 Issue（问题）

**名称**：问题（Issue）

**首次出现位置**：C01_103-仓库创建与管理.md 第1章

**定义**：
Issue 是 GitHub 用于跟踪 bug、功能请求和其他任务的模块。每个 Issue 都有标题、描述、标签、里程碑、指派人等属性。

**详解**：
Issue 不仅用于报告 bug，还可以用于功能讨论、任务跟踪、文档改进等。通过 Issue，项目管理者可以清晰地了解待办事项和进度。Issue 支持 Markdown 格式，可以添加代码块、图片等丰富内容，还可以与其他功能如 Projects、Milestones 结合使用。

---

## 2. 安全认证类

### 2.1 2FA（双因素认证）

**名称**：双因素认证（Two-Factor Authentication）

**缩写**：2FA

**首次出现位置**：C01_101-账户与安全.md 第2章

**定义**：
2FA 是一种账户安全措施，在密码之外增加第二层验证因子。登录时需要验证"你知道的"（密码）和"你持有的"（第二验证因子），大大提高账户安全性。

**详解**：
第二验证因子包括：TOTP（基于时间的一次性密码，由手机验证器 App 生成）、WebAuthn（硬件密钥如 YubiKey）、SMS（短信验证码，不推荐）、Passkey（基于 WebAuthn 的无密码方案）。安全等级排序为：硬件密钥 > TOTP > Passkey > SMS。

---

### 2.2 SSH（安全外壳协议）

**名称**：安全外壳协议（Secure Shell）

**缩写**：SSH

**首次出现位置**：C01_102-SSH与HTTPS.md 第1章

**定义**：
SSH 是一种加密的网络协议，用于安全地访问远程计算机。SSH 使用非对称密钥对（公钥/私钥）进行身份认证，是 Git 远程操作的首选方式。

**详解**：
SSH 的核心优势在于：使用 Ed25519 或 RSA 算法生成密钥对，公钥放在 GitHub 上，私钥保存在本地。连接时无需输入密码，安全性高且使用方便。常用命令包括 ssh-keygen（生成密钥）、ssh-add（添加密钥到 ssh-agent）等。

---

### 2.3 HTTPS（超文本传输协议安全版）

**名称**：超文本传输协议安全版（HyperText Transfer Protocol Secure）

**缩写**：HTTPS

**首次出现位置**：C01_102-SSH与HTTPS.md 第1章

**定义**：
HTTPS 是 HTTP 的加密版本，通过 SSL/TLS 协议对数据传输进行加密。启用 2FA 后，GitHub 不再支持账户密码进行 HTTPS 认证，需要使用 Personal Access Token 代替。

**详解**：
使用 HTTPS 方式克隆仓库时，用户名是 GitHub 用户名，密码不再是账户密码而是 PAT。配置凭据管理器（如 Git Credential Manager）可以缓存凭据，避免每次操作都输入 PAT。HTTPS 的优势在于网络兼容性好，几乎所有网络环境都能支持。

---

### 2.4 PAT（个人访问令牌）

**名称**：个人访问令牌（Personal Access Token）

**缩写**：PAT

**首次出现位置**：C01_101-账户与安全.md 第3章

**定义**：
PAT 是 GitHub 提供的访问令牌，用于替代密码进行 HTTPS 认证和 API 访问。可以精细控制权限范围，过期时间可设置，到期后需重新生成。

**详解**：
PAT 有两种类型：Fine-grained token（推荐）提供更精细的权限控制，可以精确到指定仓库和特定权限；Classic token 则是传统的令牌类型。生成 PAT 时应遵循最小权限原则，只选择必要的 scopes，并设置合理的过期时间（建议 30-90 天）。

---

### 2.5 TOTP（基于时间的一次性密码）

**名称**：基于时间的一次性密码（Time-based One-Time Password）

**缩写**：TOTP

**首次出现位置**：C01_101-账户与安全.md 第2章

**定义**：
TOTP 是一种基于时间算法的动态密码协议，由手机验证器 App（如 Google Authenticator、Microsoft Authenticator、Authy）生成，每 30 秒更新一次。

**详解**：
设置 2FA 时，用户需要使用验证器 App 扫描 GitHub 显示的二维码，将 GitHub 账户与 App 绑定。之后登录时，打开 App 获取当前显示的 6 位数字验证码输入即可。TOTP 相比 SMS 更安全，因为验证码生成在本地设备上，不依赖于运营商网络。

---

### 2.6 WebAuthn（网页认证）

**名称**：网页认证（Web Authentication API）

**缩写**：WebAuthn

**首次出现位置**：C01_101-账户与安全.md 第2章

**定义**：
WebAuthn 是 W3C 制定的 Web 标准，允许服务器使用公钥加密技术进行身份验证。是硬件安全密钥（如 YubiKey）和 Passkey 背后的核心技术。

**详解**：
WebAuthn 的工作原理是：服务器发送挑战（challenge），用户设备使用私钥对挑战进行签名，服务器使用公钥验证签名。这种方式避免了密码泄露的风险，因为私钥永远不会离开用户设备。GitHub 支持使用硬件安全密钥作为 2FA 的第二因子。

---

### 2.7 Passkey（通行密钥）

**名称**：通行密钥（Passkey）

**首次出现位置**：C01_101-账户与安全.md 第2章

**定义**：
Passkey 是基于 WebAuthn 标准的无密码认证方案，将用户的设备（如手机、电脑）转变为认证因素，用户只需使用设备的解锁方式（指纹、面部、PIN）即可完成登录。

**详解**：
Passkey 相比传统密码更加安全便捷：不需要记忆复杂密码、不受钓鱼攻击影响、不存在密码泄露问题。在 GitHub 上设置 Passkey 后，登录时只需使用设备的生物识别或 PIN 验证即可。Passkey 会同步到用户的 Apple iCloud Keychain、Google Password Manager 等服务。

---

## 3. 网络协议类

### 3.1 公钥（Public Key）

**名称**：公钥（Public Key）

**首次出现位置**：C01_102-SSH与HTTPS.md 第3章

**定义**：
公钥是 SSH 密钥对的一半，用于发送给远程服务器（如 GitHub）。公钥可以公开分发，不需要保密。

**详解**：
公钥通常保存在 `~/.ssh/id_ed25519.pub` 或 `~/.ssh/id_rsa.pub` 文件中。内容格式为 `ssh-ed25519 AAAA... comment` 或 `ssh-rsa AAAA... comment`。将公钥添加到 GitHub 后，使用对应私钥连接时，服务器可以用公钥验证私钥持有者的身份。

---

### 3.2 私钥（Private Key）

**名称**：私钥（Private Key）

**首次出现位置**：C01_102-SSH与HTTPS.md 第3章

**定义**：
私钥是 SSH 密钥对的另一半，必须严格保密，不能泄露给他人。私钥保存在本地，用于证明身份。

**详解**：
私钥文件权限必须设置为只有当前用户可读写（chmod 600）。建议为私钥设置密码短语（passphrase），增加安全性。常用命令 `ssh-add` 可以将私钥加载到 ssh-agent 内存中，避免每次操作都输入密码短语。切记：私钥一旦泄露，他人即可假冒你的身份访问 GitHub。

---

### 3.3 ssh-agent

**名称**：SSH 代理（ssh-agent）

**首次出现位置**：C01_102-SSH与HTTPS.md 第3章

**定义**：
ssh-agent 是 SSH 套件中的一个后台程序，用于管理私钥并将已解密的私钥保存在内存中，避免重复输入密码短语。

**详解**：
启动 ssh-agent 后，可以使用 ssh-add 命令添加私钥。添加后的私钥会在 agent 生命周期内保持解密状态，后续的 SSH 连接无需再次输入密码短语。Windows 系统通过服务启动，macOS/Linux 通过 eval 调用初始化。合理使用 ssh-agent 可以平衡安全性和便利性。

---

### 3.4 ssh_config

**名称**：SSH 配置文件（SSH Config）

**首次出现位置**：C01_101-账户与安全.md 第8章

**定义**：
ssh_config 是 SSH 客户端的配置文件，位于 `~/.ssh/config`，用于配置多账户连接、别名、自定义密钥文件等。

**详解**：
通过 ssh_config，可以为不同的 GitHub 账户配置独立的 Host 别名和密钥。例如配置 `github.com-personal` 和 `github.com-work` 两个别名，分别使用不同的私钥文件。这样在克隆仓库时使用对应的别名即可自动使用正确的密钥，避免账户混淆。

---

## 4. Git 操作类

### 4.1 Clone（克隆）

**名称**：克隆（Clone）

**首次出现位置**：C01_103-仓库创建与管理.md 第4章

**定义**：
Clone 是将远程仓库完整复制到本地的操作，克隆后会获得仓库的全部历史记录和文件。

**详解**：
常用命令：`git clone https://github.com/username/repo.git`。支持 HTTPS 和 SSH 两种协议。浅克隆 `--depth 1` 只需最新代码，适合大仓库加速。克隆后 git remote -v 可查看远程配置。克隆是参与项目或备份仓库的第一步。

---

### 4.2 Push（推送）

**名称**：推送（Push）

**首次出现位置**：C01_103-仓库创建与管理.md 第4章

**定义**：
Push 是将本地分支的提交上传到远程仓库的操作，使远程仓库获得本地的更新。

**详解**：
常用命令：`git push origin main` 推送主分支，`git push -u origin branch` 推送并设置上游分支。如果远程分支有新的提交，push 前需要先 pull 合并。分支保护规则可能阻止直接推送，需要通过 Pull Request 方式合并代码。

---

### 4.3 Pull（拉取）

**名称**：拉取（Pull）

**首次出现位置**：C01_103-仓库创建与管理.md 第4章

**定义**：
Pull 是从远程仓库获取更新并合并到本地分支的操作，相当于 fetch + merge。

**详解**：
常用命令：`git pull origin main`。执行时 Git 先 fetch 远程最新提交，再自动 merge 到当前分支。如果有冲突需要手动解决。推荐先 fetch 查看变更，再决定是否 merge，保持对代码更新的掌控。

---

### 4.4 Fetch（获取）

**名称**：获取（Fetch）

**首次出现位置**：C01_103-仓库创建与管理.md 第4章

**定义**：
Fetch 是从远程仓库获取最新提交到本地，但不自动合并。获取的提交保存在独立的远程分支上。

**详解**：
常用命令：`git fetch origin`。与 pull 的区别是不会自动 merge，给予开发者机会先查看远程变更再决定如何处理。可以使用 `git log origin/main` 查看远程提交历史，确认无问题后再 merge。

---

### 4.5 Merge（合并）

**名称**：合并（Merge）

**首次出现位置**：C01_103-仓库创建与管理.md 第4章

**定义**：
Merge 是将一个分支的更改合并到另一个分支的操作。Git 会自动处理简单的快进合并，对于有冲突的情况需要手动解决。

**详解**：
合并类型包括：Fast-forward（快进合并，目标分支无新提交时直接移动指针）、三方合并（双方都有新提交时创建合并提交）、冲突解决（同一文件被双方修改时需要手动选择保留内容）。建议在合并前确保工作区干净，避免意外的冲突。

---

### 4.6 Rebase（变基）

**名称**：变基（Rebase）

**首次出现位置**：C01_103-仓库创建与管理.md 第7章

**定义**：
Rebase 是将一个分支的提交在另一个分支的基础上重新应用的操作，产生更线性的提交历史。

**详解**：
命令：`git rebase main`。变基会修改提交历史（重写提交），因此不要对已推送的公共分支进行变基。变基后如果需要推送，需要 force push (`git push --force`)。变基常用于整理功能分支的提交，保持提交历史的清晰。

---

### 4.7 Remote（远程）

**名称**：远程（Remote）

**首次出现位置**：C01_103-仓库创建与管理.md 第4章

**定义**：
Remote 是指向远程仓库的引用，通常命名为 origin。用于管理与远程仓库的连接。

**详解**：
常用命令：`git remote -v` 查看远程仓库列表，`git remote add upstream URL` 添加上游仓库，`git remote set-url origin URL` 修改远程 URL。origin 是默认的远程仓库别名，但可以添加多个远程地址用于同步不同来源的代码。

---

## 5. GitHub 功能类

### 5.1 GitHub Actions

**名称**：GitHub 工作流（GitHub Actions）

**首次出现位置**：C01_204-Actions与CI.md 第1章

**定义**：
GitHub Actions 是 GitHub 提供的 CI/CD 自动化工具，可以通过配置文件定义构建、测试、部署等工作流程。

**详解**：
核心概念包括：Workflow（工作流，yml 配置文件）、Job（作业，_workflow 中的任务单元）、Step（步骤，Job 中的具体操作）、Action（动作，可复用的步骤单元）。支持触发条件如 push、pull_request、schedule 等。Actions 市场提供大量预置 action，如构建、测试、部署到各种平台。

---

### 5.2 CI/CD（持续集成/持续部署）

**名称**：持续集成/持续部署（Continuous Integration/Continuous Deployment）

**首次出现位置**：C01_204-Actions与CI.md 第1章

**定义**：
CI/CD 是一种软件开发实践，CI 指频繁地将代码变更集成到主干并通过自动化测试，CD 指将通过测试的代码自动部署到各环境。

**详解**：
CI 的好处包括：尽早发现集成问题、缩短反馈周期、提高代码质量。CD 则实现自动化部署，加快发布速度。GitHub Actions 可以完整实现 CI/CD 流程，从代码提交到自动化测试再到生产环境部署。

---

### 5.3 GitHub Pages

**名称**：GitHub 页面（GitHub Pages）

**首次出现位置**：C01_205-GitHubPages.md 第1章

**定义**：
GitHub Pages 是 GitHub 提供的静态站点托管服务，可以直接从仓库部署网站，支持 Jekyll 静态生成器。

**详解**：
用户可以通过两种方式发布 Pages：从 docs 文件夹发布（适用于项目文档）或从 main 分支根目录发布。GitHub Pages 支持自定义域名和 HTTPS。适合托管项目文档、个人博客、开源项目主页等静态内容。

---

### 5.4 Branch Protection Rules（分支保护规则）

**名称**：分支保护规则（Branch Protection Rules）

**首次出现位置**：C01_201-分支保护规则.md 第1章

**定义**：
分支保护规则用于保护重要分支（如 main），防止直接推送和强制推送，确保代码必须通过 Pull Request 审查才能合并。

**详解**：
可配置的规则包括：要求 Pull Request 才能合并、要求状态检查通过、要求审查者数量、要求签名提交、包含管理员等。合理设置分支保护规则可以提高代码质量，防止意外更改，是团队协作的重要实践。

---

### 5.5 Milestone（里程碑）

**名称**：里程碑（Milestone）

**首次出现位置**：C01_203-Issues与里程碑.md 第3章

**定义**：
Milestone 是 Issue 和 Pull Request 的容器，用于追踪达到特定目标所需的进度。可以设置截止日期。

**详解**：
一个里程碑可以包含多个 Issue，代表项目的一个阶段或目标。通过里程碑，管理者可以清晰地了解项目进度和各阶段的完成情况。例如 v1.0 里程碑可能包含 20 个 Issue，完成所有 Issue 即表示 v1.0 版本准备就绪。

### 5.6 Wiki

**名称**：维基（Wiki）

**首次出现位置**：C01_103-仓库创建与管理.md 第1章

**定义**：
Wiki 是 GitHub 仓库提供的文档功能，适合存放项目的详细文档、使用指南、设计规范等长篇内容。

**详解**：
Wiki 页面使用 Markdown 编写，可以创建多个页面并相互链接。Wiki 本身也是一个 Git 仓库，可以 clone 后在本地编辑再推送更新。Wiki 适合项目文档，但要注意不是所有访问 GitHub 仓库的人都有 Wiki 写权限。

## 6. 其他概念类

### 6.1 License（许可证）

**名称**：许可证（License）

**首次出现位置**：C01_103-仓库创建与管理.md 第1章

**定义**：
License 是开源许可证文件（如 MIT、Apache-2.0、GPL-3.0），规定了代码的使用、修改和分发条款。

**详解**：
创建仓库时可以选择合适的许可证。选择许可证会影响其他人如何使用你的代码。常见许可证：MIT（宽松，允许任何使用方式，需保留版权声明）、Apache-2.0（类似 MIT，但包含专利授权）、GPL-3.0（要求衍生作品也必须开源）。无许可证的项目默认受版权保护，他人不能合法使用。

---

### 6.2 .gitignore

**名称**：Git 忽略文件（.gitignore）

**首次出现位置**：C01_103-仓库创建与管理.md 第1章

**定义**：
.gitignore 是 Git 配置文件，用于指定不需要版本控制的文件和目录，如编译输出、依赖文件夹、日志文件等。

**详解**：
.gitignore 使用 glob 模式匹配：`/` 开头表示根目录、`**/` 匹配任意目录、`!` 开头表示取反（不忽略）。创建仓库时可以选择语言对应的 .gitignore 模板，常见的如 node_modules/、**pycache**/、*.class、dist/ 等都会被忽略。

---

### 6.3 Git LFS（大文件存储）

**名称**：Git 大文件存储（Git Large File Storage）

**缩写**：Git LFS

**首次出现位置**：C01_103-仓库创建与管理.md 第6章

**定义**：
Git LFS 是 Git 扩展，用于高效管理仓库中的大文件（如二进制资源），将文件内容存储在 LFS 服务器上，仓库只保存指针引用。

**详解**：
使用 `git lfs install` 安装后，`git lfs track "*.psd"` 追踪大文件。LFS 将大文件内容存储在远程服务器，本地仓库只保存轻量指针。克隆时通过 `git lfs pull` 获取实际文件内容。这解决了大文件拖慢仓库克隆速度的问题。

---

### 6.4 Gist

**名称**：代码片段（Gist）

**首次出现位置**：C01_102-SSH与HTTPS.md 第4章

**定义**：
Gist 是 GitHub 提供的代码片段分享服务，每个 Gist 都是一个小型的 Git 仓库，可以包含多个文件。

**详解**：
Gist 分公开和私密两种。公开 Gist 会出现在搜索结果中，类似博客帖子。创建时 `git clone git@gist.github.com:id.git` 可将 Gist 克隆到本地编辑。适合分享代码片段、配置文件、脚本等短小内容。Gist 支持版本控制，但界面更轻量简洁。

---

### 6.5 Wiki 转义

**名称**：转义（Escaping）

**首次出现位置**：C01_103-仓库创建与管理.md 第6章

**定义**：
转义是指在特殊字符前添加反斜杠 `\` 或其他方式，使其按字面意思解释，避免被 GitHub 解析为格式语法。

**详解**：
例如 Issue 中输入 `#1` 会显示为指向 Issue #1 的链接，但如果想要显示 `#1` 文本，需要写作 `\#1`。在代码块中编写 Markdown 语法时尤其需要注意转义。合理的转义确保内容准确显示而不被误解。

---

### 6.6 Collaborator（协作者）

**名称**：协作者（Collaborator）

**首次出现位置**：C01_103-仓库创建与管理.md 第5章

**定义**：
Collaborator 是对仓库有直接写权限的用户，由仓库所有者或管理员邀请添加。

**详解**：
协作者可以直接 push 到仓库的分支（除非有分支保护规则），创建和关闭 Issue，参与 Pull Request 讨论和合并。添加协作者是团队内部协作的方式，与 Fork 参与开源项目不同。协作者权限级别包括：Read（只读）、Write（推送权限）、Maintain（管理设置）、Admin（完全控制）。

---

### 6.7 Organization（组织）

**名称**：组织（Organization）

**首次出现位置**：C01_103-仓库创建与管理.md 第3章

**定义**：
Organization 是 GitHub 上用于团队协作的账户类型，可以包含多个仓库和成员，适合公司或开源团队使用。

**详解**：
组织可以创建 Team，Team 再分配不同仓库的访问权限。组织级别的设置包括：成员可见性、分支保护规则、IP 白名单等。相比个人账户，组织更适合团队协作，提供更细粒度的权限管理。
