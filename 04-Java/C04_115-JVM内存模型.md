## 1. 运行时数据区 (Runtime Data Areas)

根据 JVM 规范，Java 虚拟机在运行时将内存划分为以下几个区域：

### 1.1 线程私有区域

#### 1.1.1 程序计数器 (Program Counter Register)
- **作用**: 存储当前线程执行的字节码的行号指示器
- **特点**: 
  - 每个线程独立拥有，线程私有
  - 无内存溢出 (OOM) 风险
  - 执行 Java 方法时，记录正在执行的虚拟机字节码指令的地址
  - 执行 native 方法时，计数器值为空 (Undefined)

#### 1.1.2 Java 虚拟机栈 (Java Virtual Machine Stack)
- **作用**: 存储局部变量表、操作数栈、动态链接、方法出口等信息
- **特点**:
  - 每个线程独立拥有，线程私有
  - 生命周期与线程相同
  - 栈的大小可以固定或动态扩展
  - 栈溢出会抛出 `StackOverflowError` 异常
  - 栈扩展失败会抛出 `OutOfMemoryError` 异常

#### 1.1.3 本地方法栈 (Native Method Stack)
- **作用**: 为 Native 方法服务
- **特点**:
  - 每个线程独立拥有，线程私有
  - 与 Java 虚拟机栈类似，也会抛出 `StackOverflowError` 和 `OutOfMemoryError` 异常
  - 具体实现由虚拟机决定，有的虚拟机直接使用 Java 虚拟机栈作为本地方法栈

### 1.2 线程共享区域

#### 1.2.1 Java 堆 (Java Heap)
- **作用**: 存放所有对象实例和数组
- **特点**:
  - 所有线程共享
  - 是垃圾收集器管理的主要区域，也称为 "GC 堆"
  - 可以处于物理上不连续的内存空间，但逻辑上连续
  - 大小可以固定或动态扩展
  - 堆内存不足会抛出 `OutOfMemoryError` 异常

#### 1.2.2 方法区 (Method Area)
- **作用**: 存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据
- **特点**:
  - 所有线程共享
  - 逻辑上是堆的一部分，但在 HotSpot 虚拟机中被称为 "永久代" (Permanent Generation)
  - Java 8 及以后，永久代被元空间 (Metaspace) 取代
  - 方法区内存不足会抛出 `OutOfMemoryError: Metaspace` 异常

#### 1.2.3 运行时常量池 (Runtime Constant Pool)
- **作用**: 存储编译期生成的各种字面量和符号引用
- **特点**:
  - 方法区的一部分
  - 具有动态性，运行时也可以添加常量（如 `String.intern()` 方法）
  - 常量池内存不足会抛出 `OutOfMemoryError` 异常

### 1.3 直接内存 (Direct Memory)
- **作用**: 不属于 JVM 运行时数据区的一部分，但被 Java 程序频繁使用
- **特点**:
  - 通过 `ByteBuffer.allocateDirect()` 分配
  - 不受 JVM 堆大小限制，但受限于系统总内存
  - 直接内存分配过多会导致 `OutOfMemoryError` 异常

## 2. 堆内存结构 (Heap Structure)

### 2.1 堆内存的划分
Java 堆通常分为以下几个区域：

#### 2.1.1 新生代 (Young Generation)
- **作用**: 存放新创建的对象
- **组成**:
  - **Eden 区**: 新对象首先分配到这里
  - **Survivor 0 (S0) 区**: 经过一次 GC 后存活的对象
  - **Survivor 1 (S1) 区**: 经过多次 GC 后存活的对象
- **比例**: 通常 Eden : S0 : S1 = 8 : 1 : 1
- **特点**: 对象生命周期短，GC 频率高

#### 2.1.2 老年代 (Old Generation)
- **作用**: 存放生命周期较长的对象
- **特点**: 对象生命周期长，GC 频率低
- **对象进入老年代的情况**:
  - 经过多次 GC 后仍然存活的对象
  - 大对象（可通过 `-XX:PretenureSizeThreshold` 参数设置）
  - 新生代 Survivor 区空间不足时的对象

#### 2.1.3 永久代/元空间 (Permanent Generation/Metaspace)
- **作用**: 存储类信息、常量、静态变量等
- **特点**:
  - Java 7 及之前称为永久代
  - Java 8 及以后称为元空间，使用本地内存
  - 元空间大小受限于系统内存

### 2.2 对象的分配与晋升

#### 2.2.1 对象分配流程
1. **新对象**首先分配到 **Eden 区**
2. **Eden 区**满时，触发 **Minor GC**
3. 存活的对象被移动到 **S0 区**
4. 再次触发 **Minor GC** 时，存活的对象从 **Eden 区**和 **S0 区**移动到 **S1 区**
5. 重复上述过程，对象在 **S0** 和 **S1** 之间移动，每移动一次，年龄加 1
6. 当对象年龄达到阈值（默认为 15，可通过 `-XX:MaxTenuringThreshold` 参数设置）时，晋升到 **老年代**

#### 2.2.2 大对象处理
- 大对象直接分配到 **老年代**，避免在新生代频繁 GC
- 可通过 `-XX:PretenureSizeThreshold` 参数设置大对象阈值

## 3. 垃圾回收机制 (Garbage Collection)

### 3.1 对象死亡的判断

#### 3.1.1 引用计数法
- **原理**: 给对象添加一个引用计数器，当有引用指向该对象时，计数器加 1；当引用失效时，计数器减 1；当计数器为 0 时，对象可回收
- **缺点**: 无法解决循环引用问题，Java 虚拟机不采用此方法

#### 3.1.2 可达性分析算法
- **原理**: 从 GC Roots 出发，通过引用链遍历，能到达的对象为存活对象，不能到达的对象为可回收对象
- **GC Roots 包括**:
  - 虚拟机栈中引用的对象
  - 本地方法栈中引用的对象
  - 方法区中类静态属性引用的对象
  - 方法区中常量引用的对象
  - 活跃线程引用的对象

### 3.2 引用类型

Java 中有四种引用类型，强度依次递减：

1. **强引用 (Strong Reference)**
   - 最常见的引用类型，如 `Object obj = new Object()`
   - 只要强引用存在，对象不会被回收

2. **软引用 (Soft Reference)**
   - 用 `SoftReference` 类实现
   - 内存不足时会被回收
   - 适用于缓存场景

3. **弱引用 (Weak Reference)**
   - 用 `WeakReference` 类实现
   - 下次 GC 时会被回收
   - 适用于临时对象

4. **虚引用 (Phantom Reference)**
   - 用 `PhantomReference` 类实现
   - 最弱的引用类型，无法通过虚引用获取对象
   - 仅用于跟踪对象被回收的状态

### 3.3 垃圾回收算法

#### 3.3.1 标记-清除算法 (Mark-Sweep)
- **过程**:
  1. 标记：从 GC Roots 出发，标记所有可达对象
  2. 清除：回收所有未标记的对象
- **优点**: 实现简单
- **缺点**: 会产生内存碎片，影响后续对象分配

#### 3.3.2 复制算法 (Copying)
- **过程**:
  1. 将内存分为大小相等的两块
  2. 每次使用其中一块
  3. GC 时，将存活对象复制到另一块
  4. 清空使用过的块
- **优点**: 无内存碎片，分配效率高
- **缺点**: 内存利用率低，仅使用一半内存
- **适用场景**: 新生代，因为新生代对象存活率低

#### 3.3.3 标记-整理算法 (Mark-Compact)
- **过程**:
  1. 标记：从 GC Roots 出发，标记所有可达对象
  2. 整理：将存活对象向一端移动
  3. 清除：清理边界外的内存
- **优点**: 无内存碎片，内存利用率高
- **缺点**: 整理过程开销较大
- **适用场景**: 老年代，因为老年代对象存活率高

#### 3.3.4 分代收集算法 (Generational Collection)
- **原理**: 根据对象的生命周期，将内存分为不同代，采用不同的 GC 算法
- **新生代**: 使用复制算法
- **老年代**: 使用标记-清除或标记-整理算法
- **优点**: 结合了不同算法的优点，提高 GC 效率

### 3.4 垃圾收集器

#### 3.4.1 Serial 收集器
- **特点**: 单线程收集器，GC 时暂停所有用户线程
- **适用场景**: 客户端应用，单核 CPU 环境
- **新生代**: 复制算法
- **老年代**: 标记-整理算法

#### 3.4.2 ParNew 收集器
- **特点**: Serial 收集器的多线程版本
- **适用场景**: 服务器应用，多核 CPU 环境
- **新生代**: 复制算法
- **老年代**: 配合 CMS 收集器使用

#### 3.4.3 Parallel Scavenge 收集器
- **特点**: 关注吞吐量（吞吐量 = 运行用户代码时间 / (运行用户代码时间 + GC 时间)）
- **适用场景**: 后台计算型应用
- **新生代**: 复制算法
- **老年代**: Parallel Old 收集器（标记-整理算法）

#### 3.4.4 CMS (Concurrent Mark Sweep) 收集器
- **特点**: 关注低延迟，尽可能减少 GC 停顿时间
- **适用场景**: 响应时间敏感的应用
- **老年代**: 标记-清除算法
- **过程**:
  1. 初始标记：暂停所有用户线程，标记 GC Roots 直接引用的对象
  2. 并发标记：与用户线程并发执行，标记所有可达对象
  3. 重新标记：暂停所有用户线程，修正并发标记期间的引用变化
  4. 并发清除：与用户线程并发执行，清除未标记的对象
- **缺点**: 产生内存碎片，需要额外的内存空间

#### 3.4.5 G1 (Garbage First) 收集器
- **特点**: 面向服务端应用，关注低延迟和高吞吐量
- **适用场景**: 大内存服务器应用
- **原理**: 将堆内存划分为多个大小相等的区域 (Region)
- **过程**:
  1. 初始标记：暂停所有用户线程，标记 GC Roots 直接引用的对象
  2. 并发标记：与用户线程并发执行，标记所有可达对象
  3. 最终标记：暂停所有用户线程，修正并发标记期间的引用变化
  4. 筛选回收：选择回收价值最高的区域进行回收
- **优点**: 可预测的 GC 停顿时间，无内存碎片

### 3.5 GC 类型

#### 3.5.1 Minor GC
- **触发条件**: 新生代 Eden 区满
- **特点**: 只回收新生代，速度快
- **停顿时间**: 短

#### 3.5.2 Major GC
- **触发条件**: 老年代空间不足
- **特点**: 回收老年代，速度慢
- **停顿时间**: 长

#### 3.5.3 Full GC
- **触发条件**:
  - 老年代空间不足
  - 永久代/元空间不足
  - 显式调用 `System.gc()`
  - CMS GC 时出现晋升失败
- **特点**: 回收整个堆空间（新生代 + 老年代）
- **停顿时间**: 最长

## 4. JVM 内存参数调优

### 4.1 堆内存参数

#### 4.1.1 基本参数
- **-Xms**: 初始堆大小，默认物理内存的 1/64
- **-Xmx**: 最大堆大小，默认物理内存的 1/4
- **-Xmn**: 新生代大小，通常设置为堆大小的 1/3 到 1/2
- **-XX:SurvivorRatio**: Eden 区与 Survivor 区的比例，默认 8 (Eden : S0 : S1 = 8 : 1 : 1)
- **-XX:MaxTenuringThreshold**: 对象晋升到老年代的年龄阈值，默认 15

#### 4.1.2 老年代参数
- **-XX:PretenureSizeThreshold**: 大对象直接进入老年代的阈值
- **-XX:MaxHeapFreeRatio**: 堆空间最大空闲比例，默认 70%
- **-XX:MinHeapFreeRatio**: 堆空间最小空闲比例，默认 40%

#### 4.1.3 永久代/元空间参数
- **-XX:PermSize**: 永久代初始大小（Java 7 及之前）
- **-XX:MaxPermSize**: 永久代最大大小（Java 7 及之前）
- **-XX:MetaspaceSize**: 元空间初始大小（Java 8 及以后）
- **-XX:MaxMetaspaceSize**: 元空间最大大小（Java 8 及以后）

### 4.2 垃圾收集器参数

#### 4.2.1 收集器选择
- **-XX:+UseSerialGC**: 使用 Serial 收集器
- **-XX:+UseParNewGC**: 使用 ParNew 收集器
- **-XX:+UseParallelGC**: 使用 Parallel Scavenge 收集器
- **-XX:+UseParallelOldGC**: 使用 Parallel Old 收集器
- **-XX:+UseConcMarkSweepGC**: 使用 CMS 收集器
- **-XX:+UseG1GC**: 使用 G1 收集器

#### 4.2.2 GC 行为参数
- **-XX:+DisableExplicitGC**: 禁用显式 GC (System.gc())
- **-XX:+PrintGCDetails**: 打印详细 GC 日志
- **-XX:+PrintGCDateStamps**: 打印 GC 时间戳
- **-XX:+PrintHeapAtGC**: GC 前后打印堆信息
- **-Xloggc:gc.log**: 将 GC 日志输出到文件

## 5. 内存溢出与内存泄漏

### 5.1 内存溢出 (OutOfMemoryError)

#### 5.1.1 堆内存溢出
- **错误信息**: `java.lang.OutOfMemoryError: Java heap space`
- **原因**: 堆内存不足，无法分配新对象
- **解决方法**:
  - 增加堆内存大小 (`-Xms`, `-Xmx`)
  - 检查是否存在内存泄漏
  - 优化对象生命周期

#### 5.1.2 栈内存溢出
- **错误信息**: `java.lang.StackOverflowError`
- **原因**: 栈深度过大，如无限递归
- **解决方法**:
  - 增加栈大小 (`-Xss`)
  - 修复递归逻辑

#### 5.1.3 元空间溢出
- **错误信息**: `java.lang.OutOfMemoryError: Metaspace`
- **原因**: 元空间不足，无法加载更多类
- **解决方法**:
  - 增加元空间大小 (`-XX:MetaspaceSize`, `-XX:MaxMetaspaceSize`)
  - 检查是否存在类加载器泄漏

### 5.2 内存泄漏 (Memory Leak)

#### 5.2.1 常见内存泄漏场景
- **静态集合类**: 静态集合持有对象引用
- **监听器和回调**: 未正确移除监听器
- **内部类引用**: 内部类持有外部类引用
- **资源未关闭**: 数据库连接、文件流等资源未关闭
- **线程局部变量 (ThreadLocal)**: 线程结束后未清理

#### 5.2.2 内存泄漏检测工具
- **JVisualVM**: JDK 自带的可视化工具
- **jstat**: 命令行工具，监控 JVM 统计信息
- **jmap**: 生成堆转储快照
- **jhat**: 分析堆转储快照
- **MAT (Memory Analyzer Tool)**: 内存分析工具

## 6. 实际应用案例

### 6.1 堆内存调优案例

#### 6.1.1 案例描述
- **问题**: 应用程序在处理大量数据时出现 `OutOfMemoryError: Java heap space`
- **分析**: 堆内存不足，无法处理大量数据
- **解决方案**:
  1. 增加堆内存大小: `-Xms2g -Xmx4g`
  2. 调整新生代大小: `-Xmn1g`
  3. 使用 G1 收集器: `-XX:+UseG1GC`
  4. 设置 GC 停顿时间目标: `-XX:MaxGCPauseMillis=200`

#### 6.1.2 调优效果
- 内存溢出问题解决
- GC 停顿时间控制在合理范围内
- 应用程序性能提升

### 6.2 内存泄漏分析案例

#### 6.2.1 案例描述
- **问题**: 应用程序运行一段时间后内存占用持续增长
- **分析**:
  1. 使用 `jmap` 生成堆转储快照: `jmap -dump:format=b,file=heap.hprof <pid>`
  2. 使用 MAT 分析堆转储快照
  3. 发现大量 `HashMap` 实例持有对象引用
- **解决方案**:
  1. 检查代码中 `HashMap` 的使用，确保及时清理不需要的对象
  2. 使用弱引用或软引用存储缓存对象
  3. 定期清理缓存

#### 6.2.2 解决效果
- 内存占用稳定
- 避免了内存泄漏
- 应用程序运行更稳定

### 6.3 GC 优化案例

#### 6.3.1 案例描述
- **问题**: 应用程序 GC 停顿时间过长，影响用户体验
- **分析**:
  1. 查看 GC 日志，发现 Full GC 频繁
  2. 老年代空间不足，导致频繁 Full GC
- **解决方案**:
  1. 增加堆内存大小: `-Xms4g -Xmx4g`
  2. 调整新生代大小: `-Xmn2g`
  3. 使用 CMS 收集器: `-XX:+UseConcMarkSweepGC`
  4. 调整 CMS 参数: `-XX:CMSInitiatingOccupancyFraction=75 -XX:+UseCMSInitiatingOccupancyOnly`

#### 6.3.2 优化效果
- Full GC 频率降低
- GC 停顿时间减少
- 应用程序响应速度提升

## 7. 最佳实践

### 7.1 内存管理最佳实践
- **合理设置堆内存大小**: 根据应用程序需求和服务器资源设置合适的堆大小
- **选择合适的垃圾收集器**: 根据应用程序特点选择合适的收集器
- **监控 GC 行为**: 定期分析 GC 日志，及时发现问题
- **避免内存泄漏**: 正确管理对象生命周期，及时释放不再使用的对象
- **使用内存分析工具**: 定期使用内存分析工具检查内存使用情况

### 7.2 代码层面最佳实践
- **减少对象创建**: 避免在循环中创建大量临时对象
- **使用对象池**: 对于频繁创建的对象，考虑使用对象池
- **合理使用集合**: 选择合适的集合类型，避免过度使用 HashMap
- **关闭资源**: 使用 try-with-resources 确保资源正确关闭
- **避免无限递归**: 确保递归有终止条件
- **使用弱引用缓存**: 对于缓存场景，考虑使用弱引用或软引用

### 7.3 调优最佳实践
- **渐进式调优**: 逐步调整参数，观察效果
- **基准测试**: 在调优前后进行基准测试，评估调优效果
- **监控指标**: 关注 GC 停顿时间、吞吐量、内存使用等指标
- **根据应用特点调优**: 不同类型的应用有不同的调优策略
- **持续监控**: 生产环境持续监控 JVM 运行状态

## 8. 常见误区

### 8.1 内存调优误区
- **盲目增加堆内存**: 堆内存过大可能导致 GC 停顿时间变长
- **忽略新生代大小**: 新生代大小对 GC 性能影响很大
- **过度使用 Full GC**: 频繁的 Full GC 会严重影响性能
- **忽略元空间**: Java 8+ 中，元空间不足也会导致 OOM

### 8.2 垃圾收集器选择误区
- **认为 G1 收集器一定最好**: G1 适合大内存场景，小内存场景可能不如其他收集器
- **忽略应用特点**: 不同应用适合不同的收集器
- **不考虑硬件环境**: 收集器性能与硬件环境密切相关

### 8.3 内存泄漏误区
- **认为内存泄漏只发生在老年代**: 新生代也可能发生内存泄漏
- **忽略软引用和弱引用**: 不正确使用软引用和弱引用也可能导致内存问题
- **不及时清理资源**: 资源未关闭是常见的内存泄漏原因
